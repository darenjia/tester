# 配置系统统一设计

## 1. 背景

当前项目同时存在两套配置实现：

- `config/settings.py`
- `config/config_manager.py`

它们分别维护自己的默认值、配置文件路径、读写逻辑和访问接口，导致以下问题：

1. 同一配置项可能存在两套来源，行为不可预测
2. 热更新只覆盖部分模块，运行时状态不一致
3. 上报、执行、入口服务等主链路模块读取的并非同一份配置快照
4. 新增配置项时需要修改两处默认值和两套加载逻辑，维护成本高

第二阶段整改的目标是一次到位地统一配置系统，只保留一个真实配置实现和一个配置文件入口。

## 2. 目标

本次配置系统统一的目标如下：

1. 仓库中只保留一个真实配置实现
2. 唯一配置文件入口为 `config.json`
3. 所有运行期模块共享同一个配置快照和同一个热更新机制
4. 旧导入路径在本轮迁移中仍可工作，但内部必须转发到统一内核，不能保留第二套实现
5. 全量自动化测试通过，并同步更新文档

## 3. 非目标

本次改造不包含以下内容：

1. 任务模型统一
2. 任务队列持久化结构调整
3. 适配器抽象重构
4. 配置项语义大幅调整
5. 配置文件拆分为多环境、多层级或远端配置中心

本次只解决“单文件入口、单配置实现、全项目统一读取”的问题。

## 4. 最终方案

### 4.1 总体思路

最终只保留 `config.json` 作为唯一配置文件入口，并建立一个统一配置内核 `UnifiedConfigManager`。该内核负责：

- 默认值合并
- 配置读取与写回
- 配置快照管理
- 配置校验
- 文件热更新
- 配置变更回调

`config/settings.py` 与 `config/config_manager.py` 不再保留各自独立的 `ConfigManager` 实现。它们只允许作为导入兼容层存在，内部统一转发到同一个配置内核实例。

### 4.2 单文件入口

统一后的唯一文件入口为仓库根目录下的 `config.json`。

统一后不再读取以下文件作为真实配置源：

- `config/executor_config.json`
- 任何第二配置文件

如果未来需要环境区分，也必须建立在统一配置内核之上，而不是重新引入第二套配置实现。

## 5. 组件设计

### 5.1 UnifiedConfigManager

唯一对外配置服务，提供以下能力：

- `get(key, default=None)`
- `set(key, value, persist=False)`
- `update(config_dict, persist=False)`
- `reload()`
- `get_all()`
- `validate_config()`
- `reset_to_default()`
- `start_watcher(interval=5.0)`
- `stop_watcher()`
- `register_change_callback(callback)`

职责：

1. 从 `config.json` 加载用户配置
2. 将用户配置与统一 schema 中的默认值做深度合并
3. 持有当前唯一配置快照
4. 管理 watcher 生命周期
5. 在配置变更时分发事件

### 5.2 ConfigSchema

集中定义默认配置结构。当前散落在两处的 `DEFAULT_CONFIG` 和 `_default_config` 必须合并到这一处。

职责：

1. 维护完整默认值
2. 定义配置结构的唯一来源
3. 为验证逻辑提供基础结构依据

### 5.3 ConfigWatcher

作为 `UnifiedConfigManager` 的内部协作者存在，不单独暴露给业务层。

职责：

1. 监控 `config.json` 修改时间
2. 触发重新加载
3. 保证热更新发生在“新快照构造成功之后”

### 5.4 Facade 层

以下模块允许继续存在，但只作为兼容导入层：

- `config/settings.py`
- `config/config_manager.py`

职责：

1. 导出统一实例
2. 提供兼容旧调用方式的别名
3. 不允许再拥有独立默认值、独立配置状态或独立 watcher

## 6. 调用边界

统一后，所有业务代码都只能依赖统一配置实例。

### 允许的访问方式

- `from config.config_manager import config_manager`
- `from config.settings import settings`

以上两种入口都必须指向同一个底层实例。

### 不允许的状态

1. `settings` 和 `config_manager` 分别持有不同配置对象
2. 两个模块分别读取不同配置文件
3. 两套默认值不一致
4. 两套 watcher 同时工作

## 7. 数据流

### 7.1 启动加载

1. 统一配置内核读取 `config.json`
2. 如果文件不存在，则用默认值生成初始配置并落盘
3. 如果文件存在，则解析并与默认值深度合并
4. 生成当前配置快照
5. 业务模块通过统一实例读取配置

### 7.2 热更新

1. watcher 发现 `config.json` 发生变更
2. 读取文件内容
3. 解析 JSON
4. 运行配置校验
5. 构造新的配置快照
6. 只有上述步骤全部成功，才替换当前配置
7. 触发配置变更回调

### 7.3 写回

1. 业务调用 `set()` 或 `update()`
2. 统一实例更新内存快照
3. 如果要求持久化，则写回 `config.json`
4. 写回成功后刷新 watcher 的文件时间基线

## 8. 错误处理策略

### 8.1 配置文件不存在

行为：

1. 使用默认值构造初始配置
2. 自动写入 `config.json`
3. 记录日志说明配置文件已自动生成

### 8.2 配置文件格式非法

行为：

1. 记录错误日志
2. 优先回退到最后一次可用配置快照
3. 如果没有可用快照，则回退到默认值
4. 健康检查或状态接口应能暴露当前配置处于回退状态

### 8.3 热更新失败

行为：

1. 不污染当前运行中的配置快照
2. 保留旧配置继续运行
3. 记录失败原因
4. 等待下一次文件变更或手动 reload

## 9. 一步到位迁移顺序

### 9.1 建立统一内核

先实现统一 schema 与 `UnifiedConfigManager`，完整覆盖当前两套能力：

- 默认值
- 读写
- 校验
- 热更新
- 变更通知

### 9.2 替换主链路调用点

优先替换以下模块到统一配置内核：

- `main_production.py`
- `core/task_executor_production.py`
- `utils/report_client.py`
- API 配置相关模块
- 状态监控与配置读取模块

### 9.3 收缩兼容入口

让 `config/settings.py` 和 `config/config_manager.py` 只保留导出和转发逻辑，不再持有独立实现。

### 9.4 删除重复实现

确认调用点迁移完成后，删除：

- 第二套默认值定义
- 第二套配置加载实现
- 第二套 watcher 逻辑
- 第二配置文件入口相关代码

### 9.5 文档收尾

同步更新：

- README
- 架构文档
- 配置说明
- 部署说明

确保所有文档只指向 `config.json`。

## 10. 测试策略

### 10.1 单元测试

覆盖以下能力：

1. 默认值加载
2. 用户配置与默认值的深度合并
3. `get/set/update/get_all`
4. `reload()`
5. `validate_config()`
6. 文件不存在时自动生成
7. 文件非法时的回退行为

### 10.2 热更新测试

覆盖以下场景：

1. 配置文件变化后成功刷新
2. 配置文件变化但内容非法时不污染当前快照
3. 回调在配置真正切换后才触发

### 10.3 集成测试

重点验证以下模块读取到的是同一个统一配置实例：

- 主入口
- 任务执行器
- 上报客户端
- 配置 API

### 10.4 回归验证

必须执行：

- 全量 `pytest`
- 入口启动验证
- 配置接口读写验证
- 热更新行为验证

## 11. 风险与控制

### 11.1 风险

1. 一次切换改动面大，容易遗漏旧调用点
2. 热更新逻辑迁移后可能影响长时间运行的服务
3. 某些模块依赖旧配置路径或旧默认值语义

### 11.2 控制手段

1. 先做统一内核，再集中替换调用点
2. 用测试覆盖单元行为和主链路行为
3. 保留 facade 文件路径，降低导入路径破坏风险
4. 在删除旧实现前，通过全量搜索确认无残留独立状态

## 12. 交付标准

本子项目完成时必须满足：

1. 仓库内只剩一个真实配置实现
2. 唯一配置文件入口是 `config.json`
3. `settings` 与 `config_manager` 指向同一个底层实例
4. 不再存在重复默认值和重复 watcher
5. 全量 `pytest` 通过
6. README 与配置文档同步更新
