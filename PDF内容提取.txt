设备接入网络测试管理平台方案
1.1.
整体架构设计
本方案旨在构建一个从 Excel 数据导入、平台处理、API 传输到设备指令
下发的完整自动化流程。系统采用分层架构设计，确保数据流转的可靠性和
高效性。
1.1.1. 系统架构概述
系统整体架构分为四个核心层次：数据输入层、平台处理层、API 传输层
和设备执行层。数据从 Excel 文件开始，经过平台解析和字段分解，通过 API
接口传递给 Python 脚本，最终由 Python 脚本控制设备或 TSMaster 软件执行
指令，并将执行结果回传至平台。
1.1.2. 数据流向设计
Excel文件 → 平台上传 → 字段解析与分解 → API接口 → Python脚
本 → 设备/TSMaster → 结果回传。
整个流程形成一个完整的闭环，确保指令下发和结果反馈的实时性和准确
性。
1.2.Excel 数据导入与平台处理
1.2.1. Excel数据准备与导入
用户首先准备包含配置参数的 Excel 文件。为提高数据质量，建议采用模
板导入方式，即提供预定义的 Excel 模板，确保字段格式、数据类型与平台
要求一致。模板应包含明确的列标题，并在第二行或批注中给出校验规则和
示例。平台支持常见的文件格式（xls、xlsx），并对文件大小和单 sheet
导入进行限制。
1.2.2. 字段解析与分解流程
平台接收到上传的 Excel 文件后，启动解析流程。首先进行文件解析，判
断文件格式是否正确，避免大文件导致的内存溢出问题。解析完成后，进行
表头校验，将 Excel 列名与平台预定义的字段进行映射比对，确保字段对应
关系准确。
对于需要进行字段分解的场景（例如一个单元格内包含多个用分隔符连接
的参数），平台需调用专门的拆分逻辑。这可以通过正则表达式、字符串分
割函数或类似 Excel“文本分列”的功能实现，将复合信息拆分为独立的、
结构化的字段，为后续 API 传输做好准备。

1.4.1. COM介绍
COM 全称 Component Object Model，是微软为 Windows 平台软件提出的、
实现软件之间互操作的标准。它不会规定软件的具体实现，而是声明一种对
象模型，使得满足这种模型的对象之间能交互，这些对象通常被称为组件
（Component）。组件会实现特定的功能，而这些功能以特定的方式提供——
即接口（Interface），其他组件通过接口使用它们。另外，组件需要经过注
册（Registry），才能被其他软件发现和使用。注册后的组件向其他软件提
供服务，因此组件将作为服务端（COM Server），其他想要使用服务的作为
客户端（COM Client）。
在安装完成 CANoe 软件后，CANoe 已经在 Windows 组件服务管理器中注册了
CANoe COM Server，如果需要重新注册，可以在安装目录下（默认为
C:\Program Files\Vector CANoe 14 ） 的 Exec64 文 件 夹 中 找 到
RegisterComponents.exe，运行即手动注册。
